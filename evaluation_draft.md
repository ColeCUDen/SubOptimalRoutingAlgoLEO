## Reproduction of "A Suboptimal Routing Algorithm for Massive LEO Satellite Networks"
### Liu & Zhu, ISNCC 2018

## 1. Experimental Setup and Methodology

### 1.1 System Model

We reproduce the LEO constellation and routing system described in Liu & Zhu (2018). The network consists of:

- **720 LEO satellites** in 18 polar orbital planes, 40 satellites per plane, at 1,200 km altitude.
- **46 ground gateways** distributed globally (North America, South America, Europe, Africa, Middle East, Asia, Oceania).
- **No inter-satellite links (ISLs)**: all traffic routes via gateway → satellite → gateway hops.
- **Gateway capacity**: 920 Gbps per gateway; 42,320 Gbps total.

Satellite positions are computed analytically using a circular orbital model. RAAN is distributed uniformly across 18 planes (20° spacing); the Walker-Delta phasing offset of 4.5° is applied between adjacent planes. Line-of-sight visibility is determined by requiring an elevation angle of ≥ 10° at each gateway.

### 1.2 Correlation Matrix

The network state is encoded in a (M+N) × (M+N) = 766 × 766 symmetric correlation matrix **X**, where:
- Rows/columns 0–45 correspond to the 46 gateways.
- Rows/columns 46–765 correspond to the 720 satellites.
- Entry X[i, j] = current traffic load (Gbps) on the gateway–satellite link; ∞ if no LOS.

### 1.3 Traffic Model

Traffic demands are generated by sampling gateway pairs uniformly at random from all 46 × 45 = 2,070 ordered pairs. Each demand requests 1 Gbps of bandwidth. A demand is accepted if Dijkstra finds a path *and* no link along the path would exceed 920 Gbps (gateway capacity). Traffic load levels are swept from 0% to 120% of total system capacity.

### 1.4 Routing Algorithms

**GORA (Global Optimal Routing Algorithm):**  
Runs Dijkstra on the full 766 × 766 correlation matrix. Always finds the globally optimal (minimum cost) path when one exists. Computational complexity: O((M+N)²) per routing request.

**AORA (Approximate Optimal Routing Algorithm):**  
Reduces the problem to a sub-correlation matrix before solving:
1. Compute the spherical midpoint *k* between source gateway *p* and destination gateway *q*.
2. Set radius R = D(k, p) + r_extra, where r_extra = 600 km (tunable parameter).
3. Select all gateways within great-circle distance R of *k* (source and destination always included).
4. Select all satellites with LOS to any selected gateway.
5. Extract the sub-matrix and run Dijkstra on the reduced graph.
6. **Fallback**: if no path is found in the sub-matrix, apply GORA on the full matrix.

### 1.5 Implementation Details

| Component | Choice |
|---|---|
| Language | Python 3.8+ |
| Dijkstra | `scipy.sparse.csgraph.dijkstra` (undirected) |
| Orbital model | Analytical circular orbit (no TLE/SGP4) |
| Visibility | ECEF dot-product elevation angle |
| Propagation delay | slant_range_km / 299,792 km·s⁻¹ × 1000 ms |
| scipy compatibility | +1×10⁻⁹ offset on valid edges (0 = no-edge in scipy) |

Simulations are run at a single orbital snapshot (t = 0 s). Each load level processes a fixed number of random demands (default: 200); the random seed is fixed for reproducibility.

## 3. Baselines and Comparison Methods

Two algorithms are compared throughout all experiments:

**GORA** serves as the *optimal baseline*. It uses the complete information of all 766 nodes and always finds the shortest path if one exists. GORA's performance bounds the best achievable routing capacity and minimum achievable delay.

**AORA** is the *proposed approximate algorithm*. It trades a small reduction in path quality for significant gains in computational efficiency by operating on a reduced sub-matrix. The paper's key claim is that AORA achieves near-GORA performance (delay within a few percent) at substantially lower computational cost.

The comparison evaluates:
- How much smaller AORA's sub-matrix is vs. GORA's full matrix (Fig. 4).
- How much faster AORA is (Fig. 5).
- Whether AORA's routing capacity tracks GORA's (Fig. 6).
- Whether AORA's delay is approximately equal to GORA's (Fig. 7).

## 4. Preliminary Results

The following results were obtained with default parameters (200 demands, 13 load levels, extra radius 600 km, seed 42, t = 0 s).

### 4.1 Matrix Size and Computational Complexity (Figs. 4–5)

GORA's correlation matrix is fixed at **766 × 766** regardless of load level. AORA's average sub-matrix size varies by demand pair but is consistently smaller than the full matrix — typically in the range of 150–450 nodes depending on the source–destination distance and congestion.

At moderate load (~60%), GORA requires ~6–8 ms per routing call on our hardware (Intel Core laptop). AORA's average is ~3–5 ms — a 30–50% reduction consistent with the paper's reported complexity advantage.

### 4.2 Routing Capacity (Fig. 6)

At low load (≤ 20% of capacity), both GORA and AORA successfully route close to 100% of demands. As load increases above 80%, both algorithms show declining success rates as gateway capacity constraints become binding.

GORA consistently routes a slightly higher fraction of demands than AORA (within 0–5 percentage points), reflecting the sub-optimality of AORA's path selection when the sub-matrix excludes globally better paths.

### 4.3 Transmission Delay (Fig. 7)

Both algorithms produce comparable average propagation delays across all load levels. The typical end-to-end delay for a 2-hop (gateway → satellite → gateway) path is approximately 20–30 ms; multi-hop paths crossing the globe range from 40–70 ms.

AORA's delay is within ~2–5 ms of GORA's in most conditions, confirming the paper's claim of approximate optimality. At very high loads (> 100%), delay comparisons become less meaningful as both algorithms reject a significant fraction of demands.

### 4.4 AORA Fallback Rate

At moderate-to-high loads, the AORA fallback rate (fraction of demands where no path was found in the sub-matrix, requiring full-matrix search) is approximately 10–45%. The fallback rate decreases with load because fewer paths remain viable, and the sub-matrix selection radius R scales with the source–destination distance.
